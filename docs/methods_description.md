# Methods: Hybrid Text Classification for Propaganda Narratives

This document describes every classification method, prompt variant, and supporting
component implemented in the `hybrid-text-classification` codebase. The system targets
the SemEval / EMNLP propaganda-narrative subtasks: a hierarchical multi-label
classification problem with two top-level categories (Ukraine–Russia War — `URW`,
Climate Change — `CC`), a flat layer of *narratives* under each category, and a layer
of *subnarratives* under each narrative. The codebase implements two complementary
families of approaches:

1. **Fine-tuned encoder baselines** (`src/mDeberta/`) — supervised learning on
   ground-truth annotations.
2. **LLM-based prompt-engineered pipelines** (`src/H3Prompting/`) — zero-shot /
   few-shot classification through LangGraph state machines, with a configurable
   topology that supports single agents, multi-agent ensembles, actor–critic
   verification, self-consistency sampling, and Theory-of-Mind (ToM) arbitration.

All LLM pipelines are driven by `ClassificationConfig` (`config_loader.py`), which
takes a YAML file and produces a graph through `ConfigurableGraphBuilder`
(`graph_builder.py`). YAML configs are auto-generated by
[configs/experiments/generate_configs.py](../configs/experiments/generate_configs.py)
and stored under [configs/experiments/generated/](../configs/experiments/generated/).

## 1. Fine-tuned Encoder Baselines

### 1.1 mDeBERTa-v3 (multilingual, hierarchical multi-head)

The supervised baseline lives in [src/mDeberta/](../src/mDeberta/). It uses
`microsoft/mdeberta-v3-base` (≈278M parameters) with a custom hierarchical
multi-head classifier ([multihead_deberta.py](../src/mDeberta/multihead_deberta.py)).

| Component | Choice |
|---|---|
| Backbone | `microsoft/mdeberta-v3-base` |
| Hidden size | 768 |
| Max length | 512 tokens |
| Parent head | One linear layer over flat narrative space |
| Child heads | One linear layer per parent narrative (predicts that narrative's subnarratives + an `Other` slot) |
| Loss | Sigmoid focal loss, γ=2.0, with α-weights derived from per-class positive frequencies (capped at 10× to dampen extreme imbalance) |
| Hierarchical masking | Child losses are computed only on documents where the parent narrative is positive |
| Frozen layers | None — full fine-tuning |
| Optimiser | AdamW (HF `Trainer` default) |
| Learning rate | 2e-5 |
| Weight decay | 0.01 |
| Warm-up ratio | 0.1 (linear schedule) |
| Batch size | 8 (per device) |
| Gradient accumulation | 2 (effective batch = 16) |
| Epochs | 15 with early stopping (patience 5, threshold 1e-3 on F1-samples) |
| Threshold tuning | Grid search over [0.1, 0.9] in 17 steps; *separate* parent and child thresholds tuned on the validation split |
| Train data | [data/all-texts-unified/unified-annotations.tsv](../data/all-texts-unified/) (3,566 docs, no overlap with the dev set) |
| Train/val split | 90 / 10 stratified |
| MC-Dropout | Optional inference-time uncertainty (`--mc-dropout` keeps dropout active across runs) |

Pipeline scripts:
- [preprocess_dataset.py](../src/mDeberta/preprocess_dataset.py) — tokenises and writes
  `tokenized_dataset_hierarchical/` plus `mdeberta_artifacts_hierarchical/`.
- [train_mdeberta.py](../src/mDeberta/train_mdeberta.py) — trains with the HF `Trainer`.
- [run_inference.py](../src/mDeberta/run_inference.py) — multilingual inference with
  decoded thresholds; emits results in the SemEval TSV format.
- [run_inference_hub.py](../src/mDeberta/run_inference_hub.py) — same as above but
  pulls weights from `AWCO/mdeberta-v3-base-narratives-classifier-hierarchical`.
- [run_all_languages.py](../src/mDeberta/run_all_languages.py) — orchestrates the
  full pipeline across EN, BG, HI, PT, RU and writes manifests compatible with the
  downstream report tooling.

### 1.2 mDeBERTa "Originals-Only" ablation

[run_originals_only_experiment.py](../src/mDeberta/run_originals_only_experiment.py)
trains on the same backbone with the same hyperparameters but filters
`unified-annotations.tsv` to keep only `_ORIG_`-prefixed rows (≈1,699 native-language
documents); ≈1,699 translated-to-English rows and ≈168 `_TRANS_` rows are dropped.
Artefacts are stored in `mdeberta_artifacts_originals_only/` and weights in
`models/microsoft/mdeberta-v3-base_narratives_classifier_originals_only/`.

> Note: there is **no separate monolingual DeBERTa-v3-large English-only** training
> script in the codebase. The "DeBERTa for English" results in the paper come from
> running mDeBERTa restricted to the English subset.

## 2. LLM-based Pipelines (LangGraph)

All LLM pipelines share a common graph skeleton built in
[graph_builder.py](../src/H3Prompting/graph_builder.py); nodes are switched on or
off through the configuration. The state object
([state.py](../src/H3Prompting/state.py)) carries the input text, intermediate
predictions, validator feedback, multi-agent vote tables, ToM analysis, and
disagreement flags between nodes.

### 2.1 Common stages (always present)

```
[optional clean_text]
        │
   classify_category   ──► (Other) ──► handle_other_category ──► write_results
        │
[optional tom_analyze]
        │
   classify_narratives  (single or multi-agent)
        │
[optional aggregate_multi_agent_narratives]
        │
[optional validate_narratives → loop]
        │
[optional tom_arbitrate]
        │
[optional clean_narratives + fuzzy match]
        │
[optional disambiguate_narratives]
        │
   classify_subnarratives  (single or multi-agent, level-first per parent)
        │
[optional aggregate_multi_agent_subnarratives]
        │
[optional validate_subnarratives → loop]
        │
[optional clean_subnarratives + fuzzy match]
        │
   write_results
```

### 2.2 Methods (graph topologies)

#### A. Single-Agent Baseline
- `num_narrative_agents: 1`, `num_subnarrative_agents: 1`
- One LLM call per hierarchy level, no voting, no validator.
- Used as the reference point for every prompt-level ablation
  (`baseline_*` configs in [configs/experiments/generated/emnlp/](../configs/experiments/generated/emnlp/)).

#### B. Actor–Critic
- `enable_validation: true`, `enable_narrative_validation: true`,
  `enable_subnarrative_validation: true`.
- After each classification step a critic LLM is run with
  `create_narrative_critic_prompt` / `create_subnarrative_critic_prompt`. If
  `is_valid == false` the actor re-runs with a refinement prompt
  (`create_narrative_refinement_prompt`) that injects the critic's feedback into
  the original system prompt.
- Up to **3 retries** (`graph_builder.py:349`); after the third retry the result
  is accepted regardless.

#### C. Multi-agent Consensus ("Agora")
- `num_narrative_agents > 1`. Each agent receives the *same* prompt; agents are
  run sequentially with exponential-backoff retry (`invoke_with_retry`,
  [graph_nodes.py:48-100](../src/H3Prompting/graph_nodes.py)). Per-agent metadata
  (status, error type) is preserved.
- After classification, results are merged through one of three aggregators
  ([graph_nodes.py:888-953](../src/H3Prompting/graph_nodes.py)):

  | Method | Rule | Use case |
  |---|---|---|
  | **Intersection** | label appears in **all** agents | High precision (e.g., DeepSeek) |
  | **Majority** | label appears in **> 50%** of agents | Balanced; mandatory for weak models (GPT-5 nano, Llama via Together) |
  | **Union** | label appears in **any** agent | High recall |

  Intersection collapses to "Other" if any single agent fails — so weaker models
  must use majority/union (this is hard-wired in the project memory:
  GPT-5 nano with intersection ≈ 80–100 % "Other").

- Optional: `enable_vote_saving: true` writes per-agent votes to disk for
  later confusion / agreement analysis.

#### D. Self-Consistency
- Same code path as Agora, but conceptually a *single* model is sampled `k` times
  (rather than `k` distinct agents). Configured via
  `enable_self_consistency: true`, `self_consistency_k: 3` — `config_loader.py`
  forces `num_narrative_agents = max(2, k)` automatically.
- Naming convention in the configs: `sc_3_*` (k=3). Variants: `sc_3_intersection_*`,
  `sc_3_union_*`, `sc_3_*` (= majority by default).

#### E. Agora + Actor–Critic (hybrid)
- Both `num_narrative_agents > 1` *and* `enable_*_validation: true`.
- Aggregation is performed first; only the *aggregated* set is sent to the
  validator. Per-agent outputs are still preserved in
  `narrative_agent_metadata` for later analysis.

#### F. Theory-of-Mind Stage 1 (`enable_tom_stage1`)
- Adds a `tom_analyze` node that runs **once per document** before narrative
  classification (`create_tom_analysis_prompt`,
  [prompt_template.py:509](../src/H3Prompting/prompt_template.py)).
- The output is a JSON object with four keys cached in `state["tom_analysis"]`:
  `presuppositions` (1–3 reader beliefs assumed), `intent` (e.g., fear, trust
  erosion, moral outrage), `target_belief_change`, and `primary_mechanism`
  (`epistemic | emotional | identity | moral`).
- Cached and reused by P0T / P2 / P2nc prompts and by the ToM arbitrator.

#### G. ToM-informed Arbitration (`enable_tom_arbitration`)
- Triggered only when (1) `num_narrative_agents > 1` and (2) the multi-agent
  vote shows disagreement (i.e., *union ≠ intersection*).
- A dedicated arbiter LLM (`create_tom_arbitration_prompt`,
  [prompt_template.py:544](../src/H3Prompting/prompt_template.py)) receives the
  cached ToM analysis, all per-agent predictions, and the disagreed-label set,
  and returns a final narrative list with evidence quotes. Unanimous labels
  must be preserved by construction.
- Falls back to the aggregated set if the arbiter returns nothing.

#### H. Confusion-Aware Disambiguation (`enable_disambiguation`)
- Post-cleaning step. For each predicted label that is part of a known
  *confused pair* (loaded from a JSON file with `label_a`, `label_b`,
  `hard_negatives_a`, `hard_negatives_b`, `decision_rule`) the model is
  re-prompted with contrastive examples and the decision rule
  (`create_disambiguation_prompt`,
  [prompt_template.py:591](../src/H3Prompting/prompt_template.py)) and asked to
  decide `keep_a` / `keep_b` / both / neither. Pairs are derived from the
  bistochastic-TCM analysis in [src/analysis/bistochastic_tcm.py](../src/analysis/).

## 3. Prompt Variants (`prompt_template.py`)

Seven prompt levels are defined in `PROMPT_LEVELS`
([prompt_template.py:19](../src/H3Prompting/prompt_template.py)):

```
("P0", "P0'", "P0T", "P1", "P1nc", "P2", "P2nc")
```

Every prompt is composed from the same set of functional **blocks**, switched on
or off according to the level. The blocks are:

| Block | Description |
|---|---|
| **B1 — Expert role** | "You are an expert propaganda narrative analyst…" — included in every prompt. |
| **B2 — Task description** | Defines the goal: identify *actively promoted* narratives / subnarratives. Always present. |
| **B3 — Conservative rule** | "CRITICAL RULE: A narrative is present **only** if the text actively promotes…" — anti-over-prediction (spec S6.1). |
| **B4 — Cached ToM Stage 1 analysis** | Presuppositions / intent / target belief / primary mechanism (`_format_tom_block`, `prompt_template.py:69`). Requires `enable_tom_stage1`. |
| **B5 — General classification principles** | Loaded from `data/annotation_guideline_rules.json` → `general_principles`. |
| **B6 — Per-narrative decision rule** | Per-label rule from `narrative_rules` / `subnarrative_rules` blocks of the same JSON. |
| **B7 — BERTopic contrastive keywords** | Top-5 keywords per language (`data/bertopic_keywords.json[lang][narrative]`). |
| **B8 — Confused-pair disambiguation** | Per-domain decision rules for confused pairs (also in `annotation_guideline_rules.json`). |
| **B9 — Taxonomy + definitions + examples + instructions** | Always present; renders the available labels with their CSV-loaded definitions, examples and per-annotator instructions. |
| **B10 — Chain-of-thought scaffolding** | Step-by-step reasoning instructions ending with the JSON output schema. Always present. |
| **B11 — "Other"-handling block** | Subnarrative-only: how and when to add `…: Other`. |
| **B12 — JSON output schema** | Canonical output schema. Always present. |

Reading guide:
- **P0** is the original SIGIR baseline (no conservative rule, no annotation
  guidelines, no ToM, no keywords).
- **P0′** adds the conservative rule (B3) — pure anti-over-prediction ablation.
- **P0T** adds cached ToM (B4) but **not** annotation rules / keywords; it tests
  ToM in isolation.
- **P1** = P0′ + B5/B6/B7/B8 (full annotation-guideline package + keywords).
- **P1nc** = P1 without the conservative rule (B3).
- **P2** = P1 + cached ToM (B4) — the most context-rich variant.
- **P2nc** = P2 without the conservative rule.

The "no-conservative" (`*nc`) variants exist purely as ablations to measure the
isolated contribution of B3.

## 4. Side Implementations (Supporting Components)

These modules are not "methods" themselves but are essential plumbing used by
the pipelines. The table at the end of this section maps each component to the
methods/backbones that depend on it.

### 4.1 Text cleaning (`clean_text_node`)
- LLM-based pre-processing that strips boilerplate, navigation, cookie banners,
  CTAs, footers, etc. while preserving sentence order.
- Prompt forbids paraphrasing; the cleaned text is stored in
  `state["cleaned_text"]` and used by all downstream nodes when present.
- Toggle: `enable_text_cleaning: true`. Currently disabled in the EMNLP configs
  (raw text is fed to the classifier).

### 4.2 Label cleaning + fuzzy matching (`fuzzy_label_matcher.py`)
- After classification, predicted labels go through `clean_narratives_node` /
  `clean_subnarratives_node`:
  1. **Exact match** against the flat taxonomy (case-insensitive).
  2. **Auto-prefixing** — if the model returns `"Praise of Russia"` instead of
     `"URW: Praise of Russia"`, the category prefix is added when the prefixed
     form exists.
  3. **Fuzzy match** — `rapidfuzz.token_set_ratio` against the canonical labels
     and their prefix-stripped variants; default threshold 70.0
     (`fuzzy_threshold` config).
- `FuzzyMatchTracker` records every attempt (`exact / prefix_added / fuzzy /
  unmatched`) and writes a per-experiment JSON summary for downstream analysis.
- Critical for **Mistral** (semantically correct but syntactically variant
  labels).

### 4.3 Structured-output normalisation (`structured_output_helper.py`)
- Wraps every LLM call with `StructuredOutputWrapper`, which selects
  `with_structured_output()` natively for OpenAI / Azure / Anthropic and falls
  back to prompt-based JSON extraction for everything else.
- Three repair stages run before Pydantic validation:
  1. **`_repair_truncated_json`** — closes unclosed strings/brackets when
     `max_tokens` cuts the response off mid-object; salvages the last complete
     element.
  2. **`_normalize_json_keys`** — remaps mismatched top-level keys
     (e.g., **Together-hosted Llama-3.3-70B returns `"narratives"` for the
     subnarrative schema**; this is rewritten to `"subnarratives"` before
     validation).
  3. **`_repair_json_data`** — fills `null`s for required strings, defaults
     missing arrays to `[]`, etc.
- Without these repairs ≈ 5–15 % of Llama responses would be discarded.

### 4.4 Retry + error classification (`graph_nodes.py:48-100`)
- `invoke_with_retry` retries every LLM call up to **8 times** with exponential
  backoff (initial 5 s).
- `_classify_error` maps exceptions to typed reasons: `rate_limit`, `timeout`,
  `connection_error`, `server_error`, `content_filter`, `parse_error`, `unknown`.
  Only the first four are retried; `content_filter` and `parse_error` fail
  immediately.

### 4.5 Cost tracking (`cost_tracker.py`)
- Per-experiment, per-document accounting of API calls (model, node type,
  tokens, latency, success). Pricing tables hard-coded for `gpt-5-nano`,
  `gemini-2.5-flash`, `deepseek-chat`, etc. **Caveat:** when `run_with_config.py`
  cannot recover real token counts it inserts placeholders (input=2000,
  output=500); cost numbers should be treated as upper-bound estimates rather
  than ground truth.

### 4.6 Retrieval filtering (`retrieval_filter.py`)
- Optional pre-filter: encode every label (name + definition + example) with
  `all-MiniLM-L6-v2` and at runtime keep only the top-K labels by cosine
  similarity to the input text. Reduces the prompt label space for
  long-tail taxonomies.
- Toggle: `enable_retrieval`, `retrieval_top_k`. Not used in the final EMNLP
  experiments but is implemented and tested.

### 4.7 Taxonomy & definitions loaders (`label_info.py`)
- `load_taxonomy()` reads the canonical hierarchy.
- `load_narrative_definitions` / `load_subnarrative_definitions` read CSVs
  (`data/narrative_definitions.csv`, `data/subnarrative_definitions.csv`) with
  columns `{label, definition, example, instruction for annotator}`.
- `flatten_taxonomy` produces the flat sets used both for prompt rendering and
  for cleaning. Every narrative is automatically extended with an
  `…: Other` subnarrative for the catch-all.

### 4.8 Multi-run orchestration (`run_multi_experiment.py`)
- Runs the same configuration `n` times with deterministic seeds derived from
  a base seed (default 42). Per-run outputs go to
  `results/experiments/<exp>/run_<i>/` along with cost metrics, fuzzy-matching
  summaries and validation manifests; a top-level `summary.json` aggregates them.
- Required for bootstrap CIs and Wilcoxon / paired-t tests in
  [src/analysis/experiment_results_report.py](../src/analysis/experiment_results_report.py).

### 4.9 Vote saving and disagreement tracking
- `enable_vote_saving: true` writes per-document per-agent votes to
  `votes_*.json`. Used downstream for the bistochastic TCM and confusion-pair
  mining ([src/analysis/bistochastic_tcm.py](../src/analysis/)).
- `aggregate_multi_agent_narratives` also exposes
  `narrative_disagreement_detected` and `narrative_disagreement_labels` to
  drive the optional ToM arbitration node.

### 4.10 Configuration generator
- [configs/experiments/generate_configs.py](../configs/experiments/generate_configs.py)
  emits all experiment YAMLs from a small spec describing the cartesian
  product of `{method, model, language, prompt_level, aggregation}`.
- Methods recognised: `agora`, `agora_majority`, `agora_union`, `actor_critic`,
  `baseline`, `sc_3_*`. Models: `gpt5nano`, `gemini`, `deepseek`, `mistral`,
  `together_llama33_70b`, `hf_llama33_70b`, `deepinfra_llama33_70b`.
